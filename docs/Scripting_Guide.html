Scripting

Create, register, compile, attach and implement gameplay scripts. Scripts are small C++ classes that give objects custom behavior — the same code runs in the editor's Play mode and in the packaged game.

How scripting works

A script is a C++ class that derives from engine::Script. You override lifecycle methods, attach the script to an object by class name in the editor, and expose tunable fields. Scripts are compiled into the shared game library that both the editor and the standalone player link, so a script behaves identically in Play mode and in the shipped build.

The lifecycle
MethodWhen it runs
OnCreate()Once, when the script starts (entering Play / scene load)
OnUpdate(float dt)Once per rendered frame (variable dt) — most logic goes here
OnFixedUpdate(float dt)Once per physics step (fixed dt) — physics-coupled logic
OnDestroy()Once, before the entity/script is destroyed
Two ways to author You can write scripts in code (a header under game/, registered in the game module — full control), or use the editor's built-in Script templates and Compile & Restart for a no-leave-the-app loop. This guide covers the code path in depth and the editor path at the end.

1 · Write the script class

Create a header under game/include/game/scripts/. The minimal example (this exact one ships as Spinner) rotates its object every frame:

#pragma once
#include <engine/gameplay/Script.h>
#include <engine/ecs/Components.h>
#include <glm/gtc/quaternion.hpp>

class Spinner : public engine::Script {
public:
    void OnUpdate(float dt) override {
        if (engine::ecs::Transform* t = Transform()) {
            const glm::quat spin = glm::angleAxis(dt * 1.5f, glm::vec3(0,1,0));
            t->rotation = glm::normalize(spin * t->rotation);
        }
    }
};
  • Transform() returns this object's transform (or nullptr).
  • Self() is this entity; Registry() is the world. Both are available anywhere in the class.
  • Override only the lifecycle methods you need. If the script has a .cpp, list it in game/CMakeLists.txt; header-only scripts need nothing extra.

2 · Register it in the game module

Scripts are looked up by name at runtime. Register yours in game/src/GameModule.cpp:

// 1. include your header near the top:
#include "game/scripts/Spinner.h"

void RegisterGameModule() {
    engine::ScriptRegistry& scripts = engine::ScriptRegistry::Instance();

    // 2. register it. The name MUST match what a scene stores (the class name
    //    you'll type in the editor's Script field).
    scripts.Register("Spinner", [] { return std::make_unique<Spinner>(); });
}
The name must match The string you pass to Register("...") is the identifier the editor stores on an object and looks up at runtime. It must exactly match the Class Name you set in the Inspector (Step 4). A mismatch means the object silently has no behavior.

3 · Compile

Because scripts are C++, changes take effect after a rebuild of the game library. From the editor you don't have to leave the app:

Editor ▸ Compile & Restart (script tools)

This runs the bundled 3DGScriptCompiler, rebuilds the game module, and relaunches — the build log and status are written under build/script_compile.log / .status and surfaced in the editor. Alternatively rebuild the whole project from your normal CMake build. Either way, the compiled script now runs in Play mode and the packaged player identically.

Iteration loop Edit the script → Compile & Restart → Play. Keep scripts small so each compile is quick.

4 · Attach it to an object

  1. Select an object, open the Inspector, and Add Component ▸ Script.
  2. Set Class Name to the registered name (e.g. Spinner).
  3. Enable the script. Press P to Play — the behavior runs.

The object stores the class name (and any fields) in the scene, so it's restored on load and carried into the packaged build.

5 · Expose tunable fields

Hard-coding values (speeds, ranges, targets) is limiting. Declare fields on the object's Script component in the Inspector — each has a name, a type (Float / Int / Bool / String) and a value — and read them in the script:

void OnUpdate(float dt) override {
    float speed  = GetFieldFloat("spinSpeed", 1.5f);   // reads the "spinSpeed" field
    bool  active = GetFieldBool("active", true);
    if (!active) return;
    if (auto* t = Transform())
        t->rotation = glm::normalize(glm::angleAxis(dt*speed, glm::vec3(0,1,0)) * t->rotation);
}
ReaderField type
GetFieldFloat(name, fallback)Float
GetFieldInt(name, fallback)Int
GetFieldBool(name, fallback)Bool
GetFieldString(name, fallback)String

Now the same script drives many objects with different values, tuned per-object in the editor — no recompile to change a number.

6 · The script API

Everything below is available inside your script (protected members of engine::Script).

This object & the world

Self(), Registry(), Transform()this entity, the world, this transform
FindObject(name), FindTransform(name)look another object up by name
TryGet<T>(), Has<T>(), Add<T>(), Remove<T>()access/add/remove components on self (overloads take an entity)
SpawnEmpty(name,pos), SpawnFromObject(prototypeName,pos)create objects at runtime
DestroySelf(), Destroy(entity)remove objects

Input

IsKeyDown(key), WasKeyPressed(key)held / just-pressed this frame
IsMouseButtonDown(b), WasMouseButtonPressed(b), MouseDeltaX/Y()mouse

Triggers, collision & timers

WasTriggerEntered(e), WasTriggerExited(e), IsTriggerTouching(e)respond to trigger volumes
SetTimer(seconds, cb, repeat), Delay(seconds, cb), ClearTimer(id)schedule callbacks

Animation

PlayAnimationAction(clip), PlayMaskedAnimationAction(clip,rootBone), PlayAnimationProfile(name)one-shot / layered actions (attacks, hits)
WasAnimationEvent(name), IsAnimationActionPlaying(), IsAnimationMovementLocked()react to notifies / action state

Audio, particles, camera

PlayAudio(), PlayAudioCue(path), StopAudio(), SetAudioVolume(v) …this object's audio source or a one-shot cue
PlayParticles(), BurstParticles(n), StopParticles() …this object's particle system

Persistence & flow

7 · Worked examples

Pickup — collect on touch

class Pickup : public engine::Script {
public:
    void OnUpdate(float) override {
        // fires when the player (a trigger) enters this object
        if (WasTriggerEntered(Self())) {
            PlayAudioCue(GetFieldString("sound", "Content/Audio/coin.wav"), false);
            // ...award score / set a flag your goal checks...
            DestroySelf();
        }
    }
};

Door — open when triggered

class Door : public engine::Script {
    bool m_open = false;
public:
    void OnUpdate(float dt) override {
        if (!m_open && WasTriggerEntered(Self())) m_open = true;
        if (m_open) {
            if (auto* t = Transform()) {
                float target = GetFieldFloat("openHeight", 3.0f);
                t->position.y = glm::mix(t->position.y, target, dt * 4.0f);
            }
        }
    }
};

Key-press action

void OnUpdate(float) override {
    if (WasKeyPressed(GLFW_KEY_E)) {        // interact
        if (auto* target = FindObject("Lever")) { /* ... */ }
    }
}

Remember to register each class in GameModule.cpp and set the matching Class Name on the object.

8 · The editor template path

For a faster loop, the editor's Script component can generate a starting script from a template, let you edit the source in-app, and Compile & Restart — no manual file editing. Built-in templates:

TemplateStarting point for…
Emptya blank script (just the lifecycle stubs)
Player Movementreading input and moving the object
Door Openertrigger-driven motion
Pickupcollect-on-touch + destroy
Damage Zoneapply damage on contact

Generated scripts are written into the game module (EditorGeneratedScripts.h) and registered automatically, so they behave exactly like hand-written ones. Use templates to prototype, then hand-edit as the logic grows.

Gotchas & cheat-sheet

PitfallFix
Object has no behavior in PlayClass Name must exactly match the Register("...") name, and the script must be enabled.
Changes don't take effectScripts are C++ — Compile & Restart (or rebuild) after every edit.
Physics feels frame-rate dependentPut physics-coupled logic in OnFixedUpdate(dt), not OnUpdate.
Crash when loading a scene from a callbackUse RequestSceneLoad(...) — it's deferred to a safe point.
Field change requires recompileIt doesn't — expose it as a field and read via GetFieldFloat/Int/Bool/String.

Add-a-script checklist

  1. Header in game/include/game/scripts/, subclass engine::Script.
  2. #include + scripts.Register("Name", …) in GameModule.cpp.
  3. Compile & Restart.
  4. Add a Script component to an object; set Class Name; add any fields.
  5. P to Play.

Scripts run in the shared game module linked by both the editor and the standalone player, so what you test in Play mode is exactly what ships.
