Idle → Walk → Run
Locomotion from FBX Files

Build a fully animated character whose idle, walk and run states are driven by movement speed — using separate FBX animation files and an explicit state machine with transitions. End to end: import clips → states → transitions → play in the level.

The plan

A character controller feeds a single float parameter, Speed, into the animation graph every frame. You build three states — Idle, Walk, Run — and wire transitions that fire when Speed crosses a threshold. The clips come from separate FBX files merged onto the model.

Two things this guide relies on
  • Animation Sources — the Character Editor can merge clips from external FBX files onto your model's skeleton by bone name (the Mixamo / separate-file workflow).
  • Auto-driven Speed — the player controller (and AI) publish Speed into the controller each frame. Name the parameter exactly Speed and it drives itself.
Before you start — orientation Character rigs must be Y-up. If your model was authored Z-up (common for UE/Blender assets) it will stand upright in the editor preview but lie face-down in the scene, because one object transform drives both the mesh and the collider. Re-export the model and every animation FBX with Up = Y, Forward = -Z. Then it stands correctly with a vertical capsule. Mixamo rigs are already Y-up.

1 · Load the model

  1. Open the Character Editor and start a New character.
  2. Go to the Mesh component and pick your Skeletal Model from the Content dropdown (searchable).
  3. Assign a Material if you have one.
Skeleton must match Every animation FBX you add next must share this model's bone names. Mixamo guarantees this; a custom rig must be exported from the same skeleton.

2 · Add the animation files (Animation Sources)

Switch to the Animation component. At the top is Animation Sources (separate files).

Animation ▸ Animation Sources ▸ Add Animation File (repeat for each clip)

For each of your three files, fill the row:

File (pick from Content)Clip NameStrip Root Motion
Idle FBXIdleoff
Walk FBXWalkon
Run FBXRunon
  • File is a searchable dropdown of the FBX/glTF files in your Content folder — no typing paths. Use Refresh Files if you just imported them.
  • Strip Root Motion on Walk and Run keeps the animation in place so the character doesn't slide across the floor as the controller moves it. Leave it off for Idle.
What just happened The three clips are now merged onto your model by bone name and appear in every clip dropdown below — so from here on they behave exactly like clips that were embedded in the model file.

3 · Declare the Speed parameter

Under Parameters, add the value the graph reacts to.

Animation ▸ Parameters ▸ Add Parameter
NameTypeDefault
SpeedFloat0

This is the parameter the player controller feeds automatically at runtime (along with Direction, Acceleration, VerticalSpeed, etc. if you ever need them).

4 · Create the three states

Under States, click Add State three times and configure each:

State NameClipLoopSpeed (playback)
IdleIdleon1.0
WalkWalkon1.0
RunRunon1.0

Leave Use Blend Space 1D off on each — these are plain looping clips; the transitions (next step) switch between them. The first state you add is the starting state, so add Idle first.

Playback Speed vs the Speed parameter A state's Speed is just a playback-rate multiplier for its clip. It is not the Speed parameter — that's the graph value the transitions test.

5 · Wire the transitions (the core of this guide)

Under Transitions, click Add Transition and build these four. Together they climb Idle → Walk → Run as Speed rises and fall back as it drops.

FromToParameterCompareThresholdFade
IdleWalkSpeed>=0.50.15
WalkIdleSpeed<0.50.20
WalkRunSpeed>=3.50.18
RunWalkSpeed<3.50.20

For every transition also set:

  • Exit Time = 0 — react immediately (don't wait for the current clip to finish).
  • Interrupt Blend = on — allow a transition to fire mid-blend, so fast speed changes feel responsive.
  • Priority = 0 — only matters if two transitions can pass at once; leave default here.
Optional safety net Add Idle → Run (Speed >= 3.5) and Run → Idle (Speed < 0.5) so a sudden sprint or hard stop can skip the middle state cleanly. Give these a slightly higher Priority (e.g. 1) so they win when they apply.
"From = Any State" Leaving From as Any State makes a transition global (fires from whatever state you're in). Handy later for a jump or hit-reaction; not needed for basic locomotion.

6 · Preview it

The Character Editor preview shows your model. Scrub the Speed value in the preview's parameter controls:

  • Speed 0 → Idle.
  • Raise past 0.5 → cross-fades to Walk.
  • Raise past 3.5 → cross-fades to Run.
  • Drop it back down → falls Run → Walk → Idle.

Watch the feet: if they slide, you'll fix it in Step 8. When the transitions feel right, Save the .3dgcharacter.

7 · Alternative — a 1D blend space

Discrete states with transitions give you crisp control and are easy to reason about. If instead you want a single continuous blend (no popping between states), use a blend space:

Animation ▸ States ▸ Create 1D Locomotion

This makes one Locomotion state that blends Idle→Walk→Run along Speed using sample points (idle @0, walk @~2, run @~6). No transitions are needed — the blend is smooth by construction. It's the better choice for a standard follow-cam character; the explicit state machine above is better when you want hard, controllable switches or you're adding non-locomotion states (jump, attack) around it.

You can mix both A common setup is a Locomotion blend-space state for ground movement, plus discrete Jump/Fall states reached by transitions on VerticalSpeed.

8 · Put it in the level & drive it

  1. Get the character into the scene: Character Editor ▸ Add to Scene, or drag the saved .3dgcharacter from the Content browser. It's auto-scaled to a sensible height.
  2. Add Create ▸ Gameplay ▸ Player Start and enable its Player Controller.
  3. Set the controller's Walk Speed and Run Speed to straddle your thresholds — e.g. walk ≈ 2 (above the 0.5 Idle→Walk line), run ≈ 6 (above the 3.5 Walk→Run line).
  4. Press P to Play and move. The controller feeds Speed, so walking and sprinting climb the states automatically — no scripting.
Kill foot-sliding Two rules: (1) Strip Root Motion on Walk/Run (Step 2) so the clip animates in place while the controller does the moving; (2) make the controller's Walk/Run speeds roughly match the real speed the clips were authored at, so feet and ground move together. Nudge state Speed (playback) slightly if a clip reads a touch too fast or slow.

Troubleshooting

SymptomFix
Character is a T-pose in the sceneClips didn't merge. Confirm the animation FBX shares the model's skeleton (bone names); re-check the Animation Sources rows and clip names in the states.
Character lies face-down (collider vertical)Model is Z-up. Re-export the model and the animation FBX files as Y-up (Up = Y, Forward = -Z).
Never leaves IdleSpeed isn't rising: the parameter must be named exactly Speed, and the object needs a Player Controller (or nav agent) to drive it.
Feet slideTurn on Strip Root Motion for Walk/Run and match the controller's Walk/Run speeds to the clip speeds.
Hard pop between walk and runRaise the transition Fade (0.15–0.25s), or switch to the 1D blend space (Step 7).
Flickers between two states near a thresholdGive the up and down transitions slightly different thresholds (hysteresis), e.g. Idle→Walk at 0.6 but Walk→Idle at 0.4.

The animation controller is model-agnostic and shared by the editor preview, Play mode and the standalone player — what you tune here is exactly what runs in the packaged game.
